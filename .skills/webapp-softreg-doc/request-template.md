# 请求模板

## 标准模板

请先完整审阅这个网页应用，并至少完成三轮核查：

第一轮：梳理所有页面、导航、标签页、弹窗入口。  
第二轮：检查按钮、筛选器、表格列、图表标题、状态标签、表单字段和反馈提示。  
第三轮：验证关键交互，例如页面切换、时间切换、维度切换、搜索筛选、详情入口和弹窗提交。  

然后请先输出一份“网页核查纪要”Markdown，中间稿内容必须只来自目标网页，不要补写页面中未显示的后台细节。

接着基于核查结果生成正式软件说明书，要求如下：

1. 内容必须全部来自目标网页
2. 不要引入未验证的数据库、算法、协议、部署或权限实现
3. 先写 Markdown，再生成 Word 和 PDF
4. 最终文档必须使用严格简体中文
5. 正文必须是正式说明书写法，不能只是功能清单
6. 完成后必须自审，确认内容准确、格式准确，并且完全满足我的要求

## 无范文模板

请直接按正式送审风格处理这个网页应用，不需要等待我提供其他平台的范文。

处理流程必须是：

网页审阅 -> 事实整理 -> 正式扩写 -> 内容自检 -> Word 排版 -> PDF 渲染检查 -> 最终终审

写作要求如下：

1. 默认按成熟、厚实、正式的软件说明书风格扩写
2. 每个主要页面、模块或业务区域，都要写清用途、页面构成、主要操作、业务价值和关联关系
3. 如果页面不多，就按概览区、图表区、表格区、参数区和典型路径展开，不要生硬拆章节
4. 全文只允许写页面中已核查到的内容
5. 最终 DOCX 和 PDF 不能残留 Markdown 标记
6. 完成后必须终审，确认内容和格式完全符合要求

## 对照范文模板

请参照我提供的范文说明书风格，重新生成这个网页应用的正式说明书。

要求你重点对齐以下方面：

1. 章节结构
2. 段落密度
3. 功能说明与业务场景结合方式
4. 正式、稳定、送审化的语气

注意：

- 只能学习范文的写法和结构，不能照搬范文的产品内容
- 最终文档必须仍然只来自当前目标网页
- 完成后必须重新审阅成品，确认与范文风格和我的要求都已经贴合
