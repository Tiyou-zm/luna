# 默认正式送审风格说明

## 适用场景

在以下情形下直接使用本说明，而不必等待外部范文：

- 用户没有提供参考说明书
- 用户只要求“更专业”“更像送审稿”“不要太薄”
- 已有初稿明显偏薄，需要按正式稿重写

## 总体目标

默认把成品写成“正式、完整、可送审、段落有厚度”的软件说明书，而不是网页功能摘录。
正文必须同时满足以下要求：

- 内容来源受控，只写目标网页中核查到的事实
- 结构完整，读起来像正式文档
- 每章承担明确职责，不靠空话堆字数
- 每个模块既讲功能，也讲使用场景与业务价值

## 默认章节骨架

建议按以下结构组织正文：

1. 一、引言
2. 二、系统综述
3. 三、系统使用前提
4. 四、功能模块说明
5. 五、典型业务使用路径
6. 六、说明范围与边界

进一步展开时，优先补充：

- 编写目的
- 软件概述
- 读者对象
- 术语与缩写解释
- 设计目标
- 主要功能特点
- 功能架构简介
- 使用方式说明
- 使用关注事项

## 默认扩写粒度

每个主要页面、模块或业务区域，至少覆盖以下五类信息：

1. 模块定位：这个页面解决什么问题
2. 页面构成：页面上有哪些卡片、图表、表格、按钮、筛选器或状态区
3. 主要操作：用户在这里可以做什么
4. 业务价值：这个页面为什么重要
5. 关联关系：它与前后页面或后续动作如何衔接

推荐写法：

- 先用一段说明模块定位
- 再用一到两段说明页面构成和可见信息
- 再用一段说明操作方式和交互逻辑
- 最后用一段说明业务价值和模块关联

不要把上述五类内容压缩成一两句。

## 常见页面的扩写方式

### 1. 概览页

重点说明：

- 顶部指标卡反映哪些总体情况
- 图表区展示哪些趋势、分布或对比
- 时间维度或范围切换有什么意义
- 告警、通知、状态区如何帮助用户快速判断重点

### 2. 列表页或表格页

重点说明：

- 查询与筛选区如何缩小关注范围
- 关键字段反映哪些业务属性
- 行级动作和详情入口从哪里进入
- 该页面如何支撑管理、追踪、复核或处理

### 3. 表单、弹窗或抽屉

重点说明：

- 从哪里进入
- 可见字段如何分组
- 页面上的校验提示、必填提示或结果反馈
- 提交后如何返回，或如何影响后续页面

### 4. 图表分析页

重点说明：

- 图表的观察视角和时间维度
- 指标卡、图表和说明区如何互相补充
- 分析结果如何支撑判断、预警或决策

### 5. 仿真、优化或策略页

重点说明：

- 输入条件、参数区和启动入口
- 过程展示区与结果展示区
- 建议或结论如何形成
- 与前置分析页和后续执行页如何衔接

## 小型网页的处理方式

如果网页页面数量不多，不要为了显得“丰富”而生硬拆页。
可以改为按下列维度展开：

- 统一布局与导航
- 概览区与统计区
- 图表分析区
- 表格与详情区
- 参数设置与弹窗区
- 典型业务路径

这样既能增加文档完整度，也不会偏离真实页面结构。

## 风格要求

- 语气正式、客观、稳定
- 多用说明文句式，不写口语化碎句
- 多用“用于”“支持”“便于”“实现”“可对……进行……”等正式表达
- 少用宣传性词语，不写夸张营销语
- 不把大段正文写成项目符号堆叠

## 禁止事项

以下内容一律不要补写，除非页面中已明确展示：

- 后台技术架构
- 数据库、缓存或消息队列方案
- 算法与模型实现细节
- 外部接口或协议细节
- 权限系统内部逻辑
- 部署拓扑
- 页面中未出现的业务流程

## 薄稿判定

如果出现以下情况，说明当前稿件仍然过薄，需要继续扩写：

- 每个模块只有一两段短话
- 正文大部分只是提纲式概述
- 概览页只写“展示综合信息”，没有展开卡片、图表和告警区
- 分析页只写“支持分析”，没有展开观察维度和结果用途
- 没有写模块之间的衔接关系和典型使用路径

扩写时优先补：

- 页面构成说明
- 操作路径说明
- 业务价值说明
- 模块联动说明

不要用想象出来的技术细节去凑篇幅。
