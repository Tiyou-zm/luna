# 网页软著文档通用 SOP

## 核心原则

- 先审网页，再写文档
- 先固化事实，再做正式扩写
- 先出 Markdown，再做 Word 成品
- 先做内容自检，再做版式与渲染检查
- 每次交付前必须终审，确认内容、格式和用户要求完全贴合

## 标准流程

### 1. 明确任务边界

- 明确文档类型：软件说明书、用户手册、软著附件或功能说明
- 明确目标网页或项目范围
- 明确用户是否提供参考范文
- 明确最终交付物：Markdown、DOCX、PDF、封面、目录、截图等

### 2. 完成三轮网页审阅

第一轮：全局梳理

- 找出所有页面、导航、路由、标签页、抽屉和弹窗入口
- 记录页面命名、模块边界和主要业务区域

第二轮：细节核对

- 检查按钮、筛选器、图表标题、表格列名、状态标签、卡片和反馈文本
- 记录弹窗字段、表单提示和操作入口

第三轮：交互验证

- 验证页面切换、联动、时间范围切换、维度切换、搜索、筛选、弹窗提交和反馈
- 区分真实交互和仅用于展示的静态内容

### 3. 先写事实型中间稿

先生成一份事实整理文件，例如：

- `网页核查纪要.md`
- `功能整理说明.md`

中间稿只写：

- 页面清单
- 平台定位
- 每页主要功能
- 可见卡片、图表、表格、按钮和交互

禁止在中间稿中补写未展示的后台细节。

### 4. 选择风格基线

如果用户提供范文：

- 学结构
- 学语气
- 学段落密度
- 学功能说明与业务场景结合方式

如果用户没有提供范文：

- 直接启用 `default-formal-style.md`
- 以“送审版厚度”为最低标准

### 5. 撰写正式说明书

优先采用正式结构：

- 一、引言
- 二、系统综述
- 三、系统使用前提
- 四、功能模块说明
- 五、典型业务使用路径
- 六、说明范围与边界

每个主要模块必须写清：

- 模块定位
- 页面构成
- 主要操作
- 业务价值
- 关联关系

### 6. 完成内容自检

至少执行以下检查：

- 是否像正式说明书，而不是提纲
- 是否混入其他产品内容
- 是否遗漏重要页面、图表、表格或交互
- 是否明显偏薄
- 是否满足用户提出的“扩写”“更正式”“参照范文”等要求

### 7. 生成 Word 成品

- 备份旧版 DOCX
- 用正式样式生成干净的 DOCX
- 去除 Markdown 标记泄漏
- 添加封面和目录
- 统一标题和正文字体

### 8. 导出并做渲染检查

- 导出 PDF
- 渲染为图片
- 抽查封面、目录、前段、中段、后段和末页

重点检查：

- 标题是否正常
- 中文是否正常
- 表格是否断裂
- 正文是否溢出
- 是否有大块空白页
- 是否仍残留 Markdown 标记

### 9. 做最终终审

终审是强制步骤，不能省略。

终审时逐项确认：

- 我是否完整阅读了最终正文
- 我是否重新打开并检查了最终 DOCX 或 PDF
- 内容是否准确
- 格式是否准确
- 是否完全满足用户最新要求

如果任何一项不能确认，就不能直接交付。
